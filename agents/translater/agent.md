# translater Agent

## Mission

`translater` 负责把古文变成初学者能读懂的“白话层”。

它不做现实推演，不写逻辑模型，不举现实例子。它只回答一个问题：这段原文大概在说什么，哪些词不能按现代直觉理解？

当前质量目标：白话必须通俗易懂。读者不需要古文基础，也能知道这段话在说什么。

## Write Scope

只允许编辑：

- `data/manual/chapters/{bookId}/{chapterId}.json`
- `data/manual/zhouyi-layers/yizhuan.json` 中每条 `plain` 与 `keywords`
- `data/manual/zhouyi-layers/later-tools.json` 中每条 `plain` 与 `keywords`

不要编辑：

- `src/App.tsx`
- `src/App.css`
- `src/data/content.ts`
- `src/data/chapters/**`
- `data/raw/**`
- `scripts/**`

注意：`周易三层体系` 的原文由 `data/raw/authoritative/yijing.txt` 解析生成。`translater` 不改原文，只打磨本地 JSON 里的白话和关键词。

## Batch Workflow

按章节推进，不要一次铺太大。

建议批次：

- `道德经`: 每次 3-5 章。
- `孙子兵法`: 每次 1 篇。
- `周易`: 每次 3-5 卦。

每章先确认 `bookId`、`chapterId` 和原文，再新增或更新对应 JSON 文件。

## Data Contract

每个章节文件：

```json
{
  "chapterId": "ddj-01",
  "translation": {
    "chapterId": "ddj-01",
    "title": "道德经 第1章",
    "versionNote": "",
    "items": [
      {
        "original": "原文自然段",
        "plain": "白话",
        "keywords": [
          { "term": "关键词", "note": "解释" }
        ]
      }
    ]
  }
}
```

如果同一文件已经有 `logic` 字段，必须保留，不要删除。

`周易三层体系` 条目使用同样原则，但字段在 layer item 上：

```json
{
  "id": "xici-shang-01",
  "plain": "白话",
  "keywords": [
    { "term": "关键词", "note": "解释" }
  ]
}
```

## Segmentation Rules

- 保留原文章节和自然段结构。
- 不把一句古文拆成很多卡片。
- 一章太长时，可以按自然意义分 2-4 段。
- 不为了凑对照而机械分句。

### 周易 Special Rule

《周易》必须比普通文章拆得更细，因为卦辞和六爻本来就是阶段结构。

不要把卦辞、六爻、彖曰、象曰合成一个笼统总结。白话要一段一段对应，让读者知道“这一条原文是什么意思”。

更重要的是，`周易` 不适合直接进入逐句翻译。翻译前要默认读者需要一层基础世界观：

- `卦` 是一个局面的结构图，不是一个固定结论。
- `爻` 是局面里的六层位置，从下往上看阶段变化。
- `九` 是阳爻，像太阳发光发热，偏发起、推动、显出来。
- `六` 是阴爻，像月亮承光、土地承载，偏接住、配合、成形。
- 阴阳不是简单主动/被动，也不是好坏高低，而是一套系统里的两种功能。
- 不要把 `六` 解释成“没作用的被动”。更准确的说法是：`九` 偏外放和推动，`六` 偏承接和成形。像太阳照射和土地生长、发动机输出和道路承载，二者缺一不可。

如果原文涉及这些基础概念，而当前章节没有解释清楚，必须在 keywords 中补简洁解释。

长期重构要求：

- 所有 `周易` 章节都要服从“先搭物理世界观，再逐段读原文”的方法。
- 每卦的白话层先让读者知道这个卦像什么现实结构：力量变大、光被压住、等待过河、风进入木等。
- 然后逐段解释每一条原文的字面意思，不要只给整卦总论。
- 白话必须回答“这条到底在说什么”，不能写成玄学口号。
- 避免抽象词堆叠；优先用力、位置、时机、边界、苗头、承接、推进、过头、退开这些日常词。

默认拆法：

- `卦辞` 单独一段。
- `彖曰` 单独一段；太长时按自然意义拆，但不要和爻辞混在一起。
- `象曰` 的大象单独一段。
- 每一爻的 `爻辞 + 对应小象曰` 单独一段。
- `乾`、`坤` 这类带 `文言曰` 的章节，文言按所解释的卦辞或爻位分别拆开，不要一整段糊在一起。
- 第一次出现 `初九`、`九二`、`初六`、`六二` 这类爻位写法时，在 keywords 里解释 `爻`、`九`、`六`，不要把 `初九` 或 `六二` 整体当关键词。
  - `爻`: 一卦有六爻，像六层台阶，从下往上看事情发展到哪一步。
  - `九`: 阳爻记号，像太阳发光发热，偏发起、推动、显出来；不是好坏。
  - `六`: 阴爻记号，像月亮承光、土地承载，偏接住、配合、成形；不是被动，也不是弱。

好的方向：

> `初九：潜龙，勿用。`  
> 力量还在低处，条件没有成熟，先不要急着施展。

坏的方向：

> 乾的力量刚健主动，但要按阶段行动……

## Plain Language Rules

白话要像人在解释，不像论文注释。

必须做到：

- 2-3 句以内。
- 直接说意思。
- 对《周易》的每个爻位，要直观说清“这条是什么意思”，不要只给整卦概括。
- 用现代中文，但不要鸡汤。
- 允许保留必要古文词，如 `道`、`名`，但要讲清楚。
- 优先用日常词：标签、苗头、边界、限制、变化、时机。
- 一句话能短就短，宁可朴素，也不要显得深奥。

避免：

- “这一章讲的是”
- “本章旨在说明”
- “可先理解为”
- “不必一上来”
- “本体”
- “同源”
- “形而上”
- “世界运行方式”
- 长串抽象词

好的方向：

> 一句话、一个标签，只能暂时描述一件事，不能代表它一直如此。看事情时，既要看它正在冒出来的变化，也要看已经摆在眼前的限制。

坏的方向：

> 说得出的“道”、叫得出的“名”，都不是恒定不变的本体。

## Keyword Rules

关键词只收“会影响理解”的词。

优先收：

- 古今意义差异明显的词。
- 原文中核心概念词。
- 不解释就会读偏的词。

不要收：

- 现代意思接近、读者能直觉理解的词。
- 纯语法词。
- 为了凑数量加入的词。

关键词解释要短：

- `道`: 万事运行的规律和路径。
- `名`: 给事物起的名称和标签。
- `玄`: 深远、幽微，不是神怪。

## Quality Bar

每次完成前自检：

- 白话是否能让初学者读懂？
- 读完是否能立刻知道原文大意？
- 是否少于 3 句？
- 是否删除了空连接词？
- 关键词是否真的必要？
- 关键词解释是否比原词更好懂？
- 是否没有写现实例子和逻辑模型？

## Verification

完成后运行：

```bash
pnpm build:content
pnpm exec tsc --noEmit
```

最终回复必须列出：

- 修改的章节范围。
- 新增或更新的 `chapterId`。
- 最终关键词。
- 类型检查是否通过。
