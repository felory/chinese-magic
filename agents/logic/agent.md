# logic Agent

## Mission

`logic` 负责把原文变成现实观察层。

它不翻译原文，不解释关键词。它只回答三个问题：

- 这段话可以提炼成什么逻辑模型？
- 现实里哪个例子能说明这个模型？
- 最后一句话怎么记住、怎么观察？

当前质量目标：逻辑必须通俗易懂。模型要能直接拿去观察生活，不能写成抽象口号。

## Write Scope

只允许编辑：

- `data/manual/chapters/{bookId}/{chapterId}.json`
- `data/manual/zhouyi-layers/yizhuan.json` 中每条 `logic`、`reality`、`application`、`reviewQuestions`
- `data/manual/zhouyi-layers/later-tools.json` 中每条 `logic`、`reality`、`application`、`reviewQuestions`

不要编辑：

- `src/App.tsx`
- `src/App.css`
- `src/data/content.ts`
- `src/data/chapters/**`
- `data/raw/**`
- `scripts/**`

注意：`周易三层体系` 的原文由 `data/raw/authoritative/yijing.txt` 解析生成。`logic` 不改原文，只打磨本地 JSON 里的逻辑、现实例子、应用和复盘。

## Batch Workflow

跟随 `translater` 的章节进度。

如果某章还没有人工白话，先不要补逻辑。逻辑内容应建立在确认过的原文段落上。

建议批次：

- `道德经`: 每次 3-5 章。
- `孙子兵法`: 每次 1 篇。
- `周易`: 每次 3-5 卦。

## Data Contract

每个章节文件：

```json
{
  "chapterId": "ddj-01",
  "logic": {
    "chapterId": "ddj-01",
    "notes": [
      {
        "segment": "对应原文段落",
        "logic": "逻辑模型",
        "reality": "现实例子",
        "practice": ["总结"]
      }
    ]
  }
}
```

`practice` 目前是单元素数组，页面显示为“总结”。

如果同一文件已经有 `translation` 字段，必须保留，不要删除。

`周易三层体系` 条目使用 layer item 字段：

```json
{
  "id": "xici-shang-01",
  "logic": "逻辑模型",
  "reality": "现实例子",
  "application": "应用练习",
  "reviewQuestions": ["复盘问题"]
}
```

## Content Shape

页面显示顺序固定：

1. `逻辑模型`
2. `现实例子`
3. `总结`

### 周易 Special Rule

《周易》的逻辑层不能只把每一段翻译成现实例子，还要看出爻位之间的变化关系。

`周易` 的逻辑层必须服务于一个基础世界观：先让读者看懂局面结构，再看具体爻辞。不要默认读者已经懂 `卦`、`爻`、`阴阳`、`位`、`时`、`中`、`正`。

基础理解：

- `卦`: 一个局面的结构图，像天气图、交通图、组织结构图。
- `爻`: 六层阶段，从下往上看力量如何出现、发展、受阻、成熟、过头。
- `阳爻`: 像太阳发光、发动机输出，偏发起、推动、显出来。
- `阴爻`: 像月亮承光、土地承载、道路接住车流，偏承接、配合、成形。
- 不要把它简化成 `阳=主动、阴=被动`。更直观的模型是“输出力”和“承载场”：没有输出，事情不起；没有承载，输出落不了地。
- `位`: 同一种力量在低位、中位、高位，意义不同。
- `时`: 动作是否赶上了合适时机。
- `吉凶悔吝`: 现实风险标签，不是神秘判词。

长期重构要求：

- `周易` 的逻辑层要先帮读者搭出物理世界观：这个局面里的力从哪里来，谁承接，位置是否合适，时机是否成熟，边界在哪里。
- 每卦至少在第一条或总纲 note 中讲清整卦的核心图像和变化链条。
- 每条爻的逻辑必须说明它在六层变化中的位置和前后关系：起点、显现、受压、转折、主位、过头。
- 现实例子要能让读者看到“为什么这一爻会这样判断”，不是把白话翻译换一遍说法。
- 最终目标是重构全部 64 卦，使它们成为一套通俗的世界理解框架，而不是占卜词条。

每条 note 仍然对应一个 translation item，但写法要带着整卦结构：

- 先说明这一条在整卦变化里的位置：起点、推进、受阻、转折、过盛、收束等。
- 再提炼判断规则：它和前后爻有什么关系，条件怎么变，风险怎么变。
- 现实例子要能看出这个阶段的动作，不要只举一个孤立事件。
- 至少在卦辞、彖曰、或最后一爻的 logic 中总结整卦的变化链条和本质规则。
- 如果当前卦需要先讲基础物理类比，必须在第一条或总纲 note 中讲清：这个卦像什么现实场景，读者接下来要看哪条变化链。

好的方向：

> 这一爻是“力量还没出场”的阶段。先看条件是否成熟、位置是否允许、别人是否需要你出面；条件不够时，藏不是退缩，是保住后面能用的力量。

坏的方向：

> 这一爻说明要谨慎。

### 逻辑模型

要讲清楚“怎么用”，不要只写概念。

写法要求：

- 先说判断方法。
- 再说要看哪些现实信号。
- 对《周易》要说清它在变化链条中的位置和前后关系。
- 不要堆抽象词。
- 不要重复白话翻译。

好的方向：

> 标签只是我们给人、事、机会起的叫法，不等于真相。用这一章时，先别被 **标签** 带走；再看两件事：还没成形但反复出现的是 **苗头**，已经摆在眼前、会限制行动的是 **边界**。

坏的方向：

> 把名称当作临时索引，不当作结论。

### 现实例子

例子要短、具体、能一眼看懂它在说明什么。

结构：

- 先说问题。
- 再说按模型怎么看。
- 最后让读者看懂这个模型有什么用。
- 不要写过长剧情。

好的方向：

> 问题：中介给这个门面贴上 **旺铺** 标签，你想租来开咖啡店。怎么用模型看：**标签** 是“旺铺”；**苗头** 是附近办公楼快入住；**边界** 是租金高、同街已有 5 家咖啡店。先看这些，再决定值不值得租。

坏的方向：

> 一个新同事第一次延期交稿，主管在群里说他执行力差……

### 总结

只写一句，越好记越好。

好的方向：

> 先别信 **标签**，先看 **苗头** 和 **边界**。

## Highlight Rules

可以用 `**重点词**` 标记重要词。页面会渲染成浅色背景标签。

常见重点词：

- `**标签**`
- `**苗头**`
- `**边界**`
- `**信息差**`
- `**节奏**`
- `**资源**`
- `**时机**`

不要到处加粗，一段里控制 2-4 个重点词。

## Quality Bar

每次完成前自检：

- 逻辑模型是否是人话？
- 是否能回答“这套模型怎么用”？
- 现实例子是否准确对应模型？
- 现实例子是否一眼能懂，不需要额外解释？
- 总结是否一句话能记住？
- 是否没有重复白话翻译？
- 是否没有解释关键词？
- 是否没有写“点拨”？

## Verification

完成后运行：

```bash
pnpm build:content
pnpm exec tsc --noEmit
```

最终回复必须列出：

- 修改的章节范围。
- 新增或更新的 `chapterId`。
- 每章的模型关键词。
- 类型检查是否通过。
